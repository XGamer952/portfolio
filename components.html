Accessible Search Bar

Redesigning Nintendo's search bar to balance visual clarity, input flexibility, and accessible feedback.

The search bar is one of the most widely used UI components, yet it varies dramatically in usability and accessibility across applications. For this case study, I compared the search bars on Spotify, Amazon, and Nintendo. Each takes a distinct approach to input types, output states, and feedback.

Ultimately, I redesigned Nintendo’s search bar to adopt the clarity and robustness of Amazon’s, while incorporating some of Spotify’s live-feedback behavior—bringing together the best practices into a streamlined and accessible component.

Input Accessibility

Spotify: No explicit submit action required—typing auto-triggers search results. Great for efficiency, but may confuse users expecting to hit Enter. Keyboard focus is visually clear, and arrow key navigation is supported in filters.

Amazon: Robust autocomplete behavior. The dropdown is keyboard-navigable, and users can use arrow keys to select suggestions. Pressing Enter confirms search. Dropdown includes category picker (e.g., All, Books), increasing flexibility. Strong visual feedback on focus.

Nintendo: Simpler interface. Typing shows suggestions, but keyboard-only users must press Enter to confirm search and access filter options later. Focus state lacks strong visual cue. No keyboard shortcut to switch categories without submitting first.

Output Accessibility

  • Focus Feedback: Amazon and Spotify both include visible outlines or filled states to indicate when the search bar is active. Nintendo’s is minimal and hard to see.
  • Suggestions: Spotify and Amazon both dynamically update suggestions in real-time. Nintendo only updates suggestions after input is paused or Enter is pressed.
  • Placeholder Text: All three apps use visible placeholder text. Amazon provides the most contextual guidance.
  • Screen Readers: Amazon uses accessible labels and ARIA roles. Nintendo’s implementation is less descriptive—“search bar” is not clearly announced.
  • Focus Order: Amazon's tab order flows cleanly from category dropdown → search input → button. Nintendo’s skips over category selection entirely until after submission.

State Models

I diagrammed both the original Nintendo search bar states and my redesigned search bar states for both mouse and keyboard users. These state models helped me analyze how users interact with the component across input methods—and where breakdowns in clarity, accessibility, and flexibility occur.

Nintendo’s State Models

The default Nintendo search bar is fast but limited: there’s no way to select a category before searching, and keyboard users don’t receive strong focus feedback. There’s also no visible search button, meaning mouse users have to rely on Enter or clicking a suggestion.

My Redesigned State Models

In my redesign, I introduced a dropdown menu for selecting categories (e.g., All, Games, Hardware). This adds clarity and upfront control, but introduces an intentional tradeoff: slightly more effort for keyboard users, as they now must Tab through an additional option. However, this improves learnability and flexibility—especially for screen reader users, who can now hear the selected category before submitting.

The focus states are now clearly outlined with a red border, giving visual confirmation across input modes. I also added keyboard navigability for the dropdown and maintained Enter/Click activation options for both keyboard and mouse users.

Redesign

Final Figma redesign of the Nintendo search bar component.

  • Added a category dropdown that can be selected before submission—mirroring Amazon's flexibility
  • High-contrast red border makes focus state obvious for all users
  • Introduced a visible search button to support mouse users who may not want to press Enter
  • Designed for full keyboard accessibility—dropdown and input are Tab-able, and Enter triggers search
  • Improved clarity for screen readers with proper labeling and ARIA descriptions

This redesign favors clarity and control over minimalism—an intentional tradeoff. While the additional dropdown and focus styles introduce more steps and visual weight, they make the component far more accessible, especially for new users or those relying on assistive technologies.

Reflection

1. One of the biggest strengths I observed in Amazon’s component was its strong visual feedback and support for all input types—mouse, keyboard, and screen readers—through clearly defined focus states and ARIA roles. I made sure to integrate that idea into my redesign by adding a clear red focus border and a labeled, tabbable dropdown so all users can perceive and interact with the component meaningfully.

2. Spotify's search is efficient but inaccessible: there’s no visible search button, and auto-triggered results make it harder for screen readers to interpret what's happening. My design accounts for this by adding an explicit, visible search button that supports both mouse and keyboard input—improving clarity and control.

3. Nintendo’s component lacked the ability to select a category until after searching. This was a major accessibility gap I addressed by placing a dropdown before submission. It gives users more upfront control and benefits those who rely on assistive tech by reducing the cognitive load of backtracking and filtering post-search.

4. According to Kat Holmes’ definition, a “mismatch” occurs when a system’s design doesn’t accommodate human diversity. My redesign solves this by making the system flexible enough to match different user needs—whether you're navigating with a keyboard, screen reader, or touch.

5. A mismatch is not just a physical impairment—it’s a disconnect between how someone needs to interact and how the system lets them. By removing the reliance on Enter key submission and enabling category filtering before input, my design removes a key barrier for users with motor, cognitive, or perceptual differences.

6. For example, a user with limited hand mobility may struggle with multi-step filters after submitting—pre-selection through the dropdown reduces strain. A user relying on a screen reader can now hear the current category read aloud before confirming their search, unlike the original Nintendo flow.

7. In most interfaces, mouse and touch users are prioritized, often at the cost of keyboard and screen reader accessibility. This trend leads to designs that look clean but silently exclude users who can’t rely on visual or point-and-click cues. My redesign intentionally flips this by optimizing keyboard and screen reader support first.

8. By treating accessibility as a foundation rather than an afterthought, I’ve created a component that’s not only more inclusive but also more understandable to everyone. Learnability and predictability improved by introducing visual and auditory feedback that benefits all users—not just those with impairments.
